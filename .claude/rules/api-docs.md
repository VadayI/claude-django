# Mandatory REST API documentation (enforced)

The REST API is the only user-facing contract this repo ships (a full production frontend, if any, lives in a separate repository). So the API documentation is **mandatory and must never drift from code** ŌĆö checked by CI as a hard gate.

## What "documentation" means

1. **OpenAPI schema** ŌĆö auto-generated by `drf-spectacular` from serializers/views, committed to `docs/api/openapi.yml`. Single source of truth for the contract.
2. **Interactive UI** ŌĆö `Swagger UI` at `/api/schema/swagger/` and `Redoc` at `/api/schema/redoc/`, served by Django in dev and on staging. This replaces the old "mini-frontend for API testing": Swagger UI is a real, schema-driven, always-fresh client.
3. **Per-endpoint human notes (when needed)** ŌĆö narrative markdown in `docs/api/<resource>.md` for non-obvious things the schema can't carry (rate-limit reasoning, deprecation path, business invariants, idempotency keys).

## The gate ŌĆö `scripts/check_openapi_drift.sh`

CI regenerates the schema from the live code and diffs it against the committed `docs/api/openapi.yml`. If they differ, the PR fails. This **physically prevents code and docs from drifting**, the same way `check_stubs.sh` prevents unlogged stubs.

Run locally before pushing:

```bash
bash scripts/check_openapi_drift.sh
# or regenerate the committed schema:
cd backend && python manage.py spectacular --file ../docs/api/openapi.yml --format openapi-yaml
```

## Lifecycle (per feature)

1. `api-architect` fixes the endpoint contract ŌåÆ declares serializer fields, response codes, permissions.
2. `tester` writes the failing API feature test (DRF `APIClient`) per the contract.
3. `django-developer` implements until GREEN ŌĆö `drf-spectacular` annotations (`@extend_schema`, `@extend_schema_field`) added where the schema needs help (custom return shapes, error formats, examples).
4. **Before opening the PR** (or in `/wrap-up`): regenerate `docs/api/openapi.yml` and commit it. The drift gate will pass.
5. `docs-writer` adds the per-endpoint narrative markdown if needed and includes the endpoint in `docs/api/INDEX.md`.

## Required `drf-spectacular` setup

- In `INSTALLED_APPS`: `drf_spectacular`.
- In `REST_FRAMEWORK`: `'DEFAULT_SCHEMA_CLASS': 'drf_spectacular.openapi.AutoSchema'`.
- `SPECTACULAR_SETTINGS` with `TITLE`, `DESCRIPTION`, `VERSION` set per project.
- URLs include `SpectacularAPIView`, `SpectacularSwaggerView`, `SpectacularRedocView`.
- `pyproject.toml` already pins `drf-spectacular>=0.27`.

## Binds these agents (rule is auto-loaded)

- `api-architect` ŌĆö the endpoint contract is incomplete until the OpenAPI fragment (serializers + `@extend_schema` annotations) is sufficient to produce a correct schema.
- `django-developer` ŌĆö adds `@extend_schema` annotations when the generator can't infer the shape; runs spectacular locally to confirm; commits the regenerated `docs/api/openapi.yml`.
- `docs-writer` ŌĆö owns `docs/api/INDEX.md` and any per-endpoint narrative; verifies the gate passes before declaring the PR ready.
- `reviewer` ŌĆö at Quality Gate, blocks PRs where the schema diff suggests breakage (renamed fields, removed endpoints, changed response codes) without an ADR / versioning step.

> Goal: the contract is alive in code and re-derived for docs on every build, so the schema is **never** out of date.
